# Victimas RD project context

## Status and purpose

This document is the required starting point for work in this repository. It
distills the Dropbox research-archive handoff inventoried on 2026-07-23.

The project studies Peru's Collective Reparations Program (`Programa de
Reparaciones Colectivas`, PRC), asking whether community-level reparations were
only transitional or produced longer-term transformational effects. The
current paper focuses on policy feedback, social protection, and migration.

The empirical strategy is a fuzzy regression discontinuity design around the
B-C threshold of a government victimization index. Administrative and census
records are linked at the community (`centro poblado`, or `ccpp`), household,
and individual levels.

This repository is an initial reproducibility scaffold. It is not evidence that
the current analysis can already be reproduced, and it is not an authoritative
replication release.

## Source handoff

The detailed source is `PROJECT_CONTEXT_FOR_GITHUB.md` at the root of the local
Victimas RD Dropbox archive. It describes the research design, existing Stata
programs, data and document inventory, current outputs, discrepancies, risks,
and a staged migration plan.

The handoff originally proposed a hyphenated clone directory. The active
repository is named `VictimasRD` without a hyphen. Code must use the local path
configuration instead of assuming either directory name.

## Separation of responsibilities

### Dropbox is authoritative for

- raw source data;
- large working, coded, and linked datasets;
- restricted community, household, and individual records;
- testimony and authorization materials;
- literature PDFs and third-party archives;
- administrative, personal, and submission documents;
- historical code and outputs not yet migrated with provenance; and
- archival outputs.

Original Dropbox data is immutable. Dropbox must not be converted into or
treated as this Git repository.

### Git is authoritative for

- reproducible Stata source code;
- documentation and decision records;
- metadata that contains no research observations;
- data contracts, tests, and validation rules;
- software and dependency manifests;
- merge, sample-flow, specification, and output manifests; and
- appropriately sized, non-sensitive tables and figures generated by versioned
  code.

Raw or restricted data must never enter Git, including Git LFS or repository
history.

## Archive inventory and existing pipeline

At the inventory date, Dropbox held approximately 75.5 GB of data plus
literature, legacy code, documents, presentations, and archived outputs. The
main data areas are:

- `2 data/1 Raw`: original administrative, census, survey, and spatial inputs;
- `2 data/2 Working`: very large cleaned and linked intermediates; and
- `2 data/3 Coded`: multiple community, household, and individual analysis
  variants.

Potentially sensitive content includes person- and household-level
administrative records, small-area data, testimony material, authorizations,
private correspondence, and personal documents. Literature and third-party
replication files may also be non-redistributable.

The most relevant legacy Stata branch appears under the Dropbox working-paper
code folder. It contains a nominal master, source-specific cleaners, central
data preparation, descriptive analysis, RD validity checks, and community-,
household-, and individual-level regressions. The legacy master is not an
end-to-end reproducible entry point: it has stale absolute paths, disabled
switches, incomplete orchestration, dynamic dependencies, and inconsistent
output handling.

Do not migrate or refactor this code casually. First preserve an exact,
auditable legacy snapshot in a dedicated task, with file-level provenance and
checksums, then build a clean modular pipeline separately.

## Critical research decisions

The following issues affect the estimand, sample, or reported results and must
be resolved by the research team before an authoritative release:

1. **Geographic sample:** the manuscript narrative includes Ayacucho areas,
   while the current executable rule appears to exclude Ayacucho.
2. **Victimization cutoffs:** official methodology, current code, and the
   January 2026 manuscript report different category boundaries.
3. **Treatment timing and event:** the code often uses financing by 2013,
   while paper and presentation language sometimes refers to 2012; approval,
   financing, initiation, and completion are distinct events.
4. **Manuscript source:** the latest apparent manuscript is a January 30, 2026
   PDF, but its complete editable source and bibliography have not been found.
5. **Canonical datasets:** several base, modified, SAT/PBI, and conflict-copy
   variants coexist without a complete lineage map.
6. **Community universe and sample flow:** community counts and the linked
   individual analytical sample differ across code, data, and manuscript.
7. **Migration measurement:** absence from a later linked record may reflect
   migration, death, linkage failure, code changes, coverage, or enumeration
   differences.
8. **Output provenance:** current manuscript claims appear to rely on older,
   multi-specification workbooks that may include manual selection or edits.
9. **Fuzzy matches:** community-name matching and manual corrections need a
   versioned accepted/rejected match ledger.
10. **Governance:** data-use, ethics, retention, licensing, access, and
    disclosure rules require definitive documentation.

Other important analysis risks include small effective samples, many outcomes
and specifications, limited cluster counts, a discrete or heaped running
variable, use of `masspoints(off)`, post-treatment mechanism variables,
spillovers, and possible sample selection based on first-stage strength.

## Reproducibility target

The desired audit chain is:

```text
documented source
    -> immutable input
    -> versioned transformation
    -> declared analysis sample
    -> pre-identified specification
    -> reproducible table or figure
    -> exact manuscript claim
```

A clean Stata pipeline should eventually cover:

1. environment and dependency validation;
2. immutable-source cleaning into a non-raw build area;
3. deterministic construction of analysis products;
4. descriptive analysis;
5. RD first-stage and validity checks;
6. community, household, and individual estimation;
7. migration and exploratory mechanism analysis;
8. final tables and figures;
9. disclosure checks; and
10. output-manifest updates.

Minimum contracts should test input presence and identity, key uniqueness,
declared merge cardinality, matched and unmatched counts, treatment dates,
official cutoff use, geographic restrictions, sample flow, first-stage
tolerances, expected diagnostic outputs, locked-result tolerances, and absence
of row-level or disclosive material in outputs.

## Canonical terminology

Prefer descriptive names such as:

| Concept | Preferred label |
|---|---|
| Community / centro poblado | `ccpp` |
| Household | `household` |
| Individual/person | `individual` |
| Victimization index | `victimization_index` |
| B-C centered running variable | `running_bc` |
| Funding or approval year | `funding_year`, or a more explicit event name |
| Treatment by a selected year | `treated_by_YYYY` |
| Main RD sample | `sample_main_rd` |
| 2007 census baseline | `census_2007` |
| SISFOH source | `sisfoh_YYYY` after its timing is resolved |
| 2017 census outcome | `census_2017` |

Avoid opaque suffixes such as `mod`, `both`, `newvariables`, or numbered copies
unless a registry explains them.
